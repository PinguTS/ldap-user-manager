# Internationalization (i18n)

The app uses **JSON locale files** under [`www/locales/`](../../www/locales/) and resolves the active language per request. **English** (`en.json`) is the canonical catalog; other files (e.g. `de.json`) only override keys they define.

## Behavior

1. If a request contains a valid explicit `lang` choice, it is used and persisted to a cookie.
2. Otherwise, if a valid locale cookie exists, it is used.
3. Otherwise, parse `Accept-Language` (RFC-style ranges and `q` weights). Each range is reduced to its **primary subtag** (`de-DE` → `de`).
4. Pick the **first** preferred language that has a matching `*.json` file in `www/locales/`.
5. If none match, use **`en`**.
6. Load `en.json`, then merge the chosen locale file on top so missing keys stay in English.

## Language chooser UI

- The chooser is rendered in the top-right navbar and on the login page.
- Each language option uses:
  - a native language label (for example, `Deutsch`, `Nederlands`, `Svenska`), and
  - a flag icon from `www/assets/flags/`.
- Flag filenames are mapped by locale code (for example, `de` -> `de.svg`).
- Invalid `lang` values are ignored safely.

## Adding a language

1. Add `www/locales/<code>.json` where `<code>` is a two-letter tag or `xx-yy` (e.g. `de`, `zh-cn`). Only keys you translate need to appear; the rest fall back to `en.json`.
2. No config change is required; the new file is picked up automatically.

### Bundled locale files

| File | Language |
|------|----------|
| `en.json` | English (canonical, full catalog) |
| `de.json` | German |
| `es.json` | Spanish |
| `fr.json` | French |
| `it.json` | Italian |
| `ja.json` | Japanese |
| `ko.json` | Korean |
| `zh.json` | Chinese (Simplified) — matches browser primary tag `zh` (e.g. `zh-CN`, `zh-SG`) |
| `hi.json` | Hindi |
| `nl.json` | Dutch |
| `sv.json` | Swedish |
| `nb.json` | Norwegian (Bokmål) |
| `no.json` | Norwegian — same strings as `nb.json` (some clients send primary tag `no`) |
| `da.json` | Danish |

**Note:** Browsers send a **primary** language subtag (`zh-CN` → `zh`). Simplified Chinese is served via `zh.json`. For Traditional Chinese (`zh-TW`) the primary tag is still `zh`, so it would currently select `zh.json`; supporting both would require extending locale matching (e.g. `zh-tw.json` + parser changes). Norwegian may appear as `nb` or `no`; both `nb.json` and `no.json` are provided (identical Bokmål strings).

## Using strings in PHP

After [`web_functions.inc.php`](../../www/includes/web_functions.inc.php) is loaded, **`t('key')`** returns the translated string. For dynamic segments use placeholders: store `"Hello, :name"` in JSON and call `t('greet', ['name' => $user])`.

Always **escape for HTML** when outputting: `htmlspecialchars(t('key'), ENT_QUOTES, 'UTF-8')`. Modal bodies that intentionally contain HTML may use `t('key')` without escaping when the JSON is trusted.

**API endpoints** under `www/api/` and **export/integration scripts** (e.g. `www/export/organizations.php`) are not part of this i18n layer.

## Caching / CDN

If HTML is cached in a shared cache without varying by language, users may see the wrong language. In that case send **`Vary: Accept-Language`** (or avoid caching authenticated HTML).

## Tests

[`tests/I18nTest.php`](../../tests/I18nTest.php) covers Accept-Language parsing, locale choice, merge behavior, and placeholders. `lum_i18n_bootstrap()` accepts an optional second argument (absolute path to a locales directory) for tests only.

Run PHPUnit with **`make test`** (uses Docker `php:8.2-cli` if PHP is not installed locally); run **`make install`** first so `vendor/bin/phpunit` exists.
